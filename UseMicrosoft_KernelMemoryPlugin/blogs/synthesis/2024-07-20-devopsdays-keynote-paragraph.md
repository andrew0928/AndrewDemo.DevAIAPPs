## 段落1, 寫在前面
本篇文章以作者參與 DevOpsDaysTaipei 的 Keynote 演講作為背景，說明他近年來在軟體開發中從 API First 轉移到 AI First 的觀點。作者先總結過去在各屆大會中分享的主題：從大型團隊的 CI/CD 挑戰、API First 的演進，一直到架構師工作中遇到的 DevOps 持續交付等議題。除了延續以往對 API 設計與落地實務的探討，這次特別著重在生成式 AI 帶來的革命性改變。作者點出，業界在模組化服務、算力與工具等層面雖有快速發展，但多數人僅停留在使用 ChatGPT、Stable Diffusion 等工具，而非將其深度整合到自家服務。寫成文章的動機在於：未來 AI 技術若更普及，每個環節都會重塑既有的思考與設計方式，因此開發者應及早認清軟體開發基礎的重要。作者同時提及幾篇先前撰寫的部落格文章，並為此次演講準備了一系列 DEMO，試圖以電子商務及零售場景案例為引，引導讀者思考如何將 AI 視為核心元件來重新打造應用程式，並借此補足先前演講沒能講到的實務串聯內容。此篇文章為 Keynote 演講的擴充與紀錄，將帶領讀者從更高層次的觀點來看 AI 如何協助軟體開發，以及多人協作時該如何在基礎面打好根基。

## 段落2, 示範案例: 安德魯小舖 GPTs
作者以實驗性專案「安德魯小舖 GPTs」為核心示範，藉由 GPTs 能夠呼叫後端 API 取得商品及結帳資訊，實現自然語言對話購物。首先，作者透過 DEMO #1 展示「出一張嘴就能買東西」的情境：使用者僅需以自然語言描述想買什麼、想花多少預算，系統便能幫忙運算並自動完成結帳流程。接下來的 DEMO #2 進一步利用 AI 推理、記錄顧客個人喜好和對商品的滿意度，讓「店長 GPT」能辨識對話中隱含的情感，並自動形成會員標記與評分。作者強調，這種「降維打擊」的使用者體驗並非要取代原先的 UI/UX 設計，而是開闢另一條把 AI 帶進產品的道路。也就是說，AI 不再只是「提供資訊」或「做出推薦」，還能透過自然語言推理和對應動作來延伸實際服務。如果能設計出良好的 API 並搭配高品質 Prompt，就能讓 AI 轉譯各種任務給執行端，因而大幅提升使用者體驗的深度與豐富度。文中亦談到，AI 的不確定性同時也是風險，開發人員需評估費用、效能與錯誤管控等面向。

## 段落3, 軟體開發還是你想像的樣貌嗎?
進入第三部分，作者將前述案例歸納到「當 AI 成為應用開發主要元件之一，整個開發模式會更貼近 AI First」的觀點。首先，他呼籲大家重視 API First 的設計品質，包括狀態機、SCOPE、認證授權邏輯與跨服務交易等，並指出「虛應故事」或過度為單一前端場景調整的 API 可能導致無法重複利用、維運困難。尤其 AI 本身善於語意推理，但不擅長精確邏輯，若 API 設計品質不佳，反而得花大量 Prompt 引導，結果浪費 token 與開發心力。作者接著深入探討「Prompt Engineering」的重要性：無論是簡單的輸入模式、Json Mode 或 function calling，開發者都需掌握如何把需求拆解成 LLM 最擅長的文字敘述部分，以及哪些工作務必保留給傳統程式執行，以求精準穩定。再者，作者用 Copilot 的模式取代純聊天框架，例如結合 .NET Console 與 Semantic Kernel，讓使用者仍可透過傳統 UI 操作系統，但在遇到不足或困難時再呼喚 AI 提供「軟性判斷」；此舉兼顧實際效能與 AI 便利性。最後，作者以 RAG（檢索增強生成）案例補充 AI 應用技巧，說明如何整合向量索引與檢索查詢，將外部知識提供給大模型增強推理能力，也點出資料、算力與部署都是 AI 時代的新挑戰，開發者需在基礎建設與 Pipeline 中納入相應策略。

## 段落4, Ref: 零售業的 AI 應用情境
為串接實際商業場景，作者引用上司 Happy 在「生成式 AI 年會」上提出的零售業 AI 應用實例。對零售業而言，售前、售中、售後都包含大量與顧客心理連結的互動；若能以 AI 為軸心建構新的代理人 Agent，並搭配後端「Engine」進行推斷或運算，將可掌握顧客隱藏的意圖或喜好，從而達成個人化銷售和更緊密的顧客關係。作者延續「安德魯小舖 GPTs」的概念，說明店家端 Agent 如何主動檢索或結合推薦、智慧手段來驅動銷售，並且配合零售業更廣泛的應用需求，如實體店鋪、全通路行銷與多種顧客觸點。這些場景都可藉由 AI Agent 與多種具備專業功能的引擎及 API 彼此串接，最終形塑出未來零售業「智慧銷售」的新典範。文中也提到未來 AI 可能扮演的是雙方代理人的角色：它同時代表消費者與店家對話，並穿插後台系統服務。在此結構下，AIDX 與服務抽象化成為關鍵，因為只有高品質、能夠安全擴充的 API 設計，才能容納 AI 多層次、動態的邏輯運用。

## 段落5, 寫在最後
文章最後做了幾項總結：第一，作者強調 AI 只是開端，未來當算力更進一步降價，開發者就能用更成熟的模型來完成更複雜抽象層次的工作；因此現在投資在高品質 API、完善的 DevOps Pipeline 與資料資產管理上，將能在未來獲得更大回報。第二，思考軟體設計時需改變單純前後端 MVC 的思維，將 LLM 作為智慧解析層，看什麼時候該以 Copilot 形式輔助使用者，或嘗試全權交給 Agent 處理。第三，要認清 AI 不可能完全取代「控制器」的精確運算，開發者必須在基礎邏輯的掌控（計算）與高度彈性判斷（推理）之間取得平衡。最後，作者反覆呼籲觀察並評估新技術如何契合產品需求，不論是零售業或其他領域，都能從先行嘗試實驗性 PoC，或精進基礎軟體開發流程做起。更進一步的落地方案，如資料部署與算力管理，皆需要新一代 AI 流程與傳統的 GitOps、CI/CD 流程共存。文章結尾也引用大會問券回饋及聽眾心得，有好有壞，但都顯示 AI 與 API 之間的相互支撐確實是未來發展的趨勢。
